# 创意记录软件 — 完整产品方案

**版本：** v1.1
**状态：** 设计完善，待开发
**性质：** 开源、本地优先


## 第一部分：产品定义

### 1.1 一句话定位

**创意过程的可视化Git客户端——从思考到实现的工作台。**

### 1.2 产品不是

| 不是 | 区别 |
|:---|:---|
| 笔记工具（Notion, Obsidian） | 不优化"记录和整理已有知识"，而优化"从模糊想法中生长出清晰方案" |
| 白板工具（Miro, FigJam） | 节点是AI辅助生长的，不是完全手动放置的 |
| AI聊天工具（ChatGPT, Claude） | 对话是手段，脉络图才是持久化的、可管理的结果 |
| 项目管理工具（Jira, Linear） | 不管理执行过程，只管理"想清楚"的过程 |

### 1.3 核心隐喻

**产品是一个"思想的堆积场"与"创意生长的腔体"。**

- **Aha模式下：** 它是一个永远不会满的抽屉。用户无需思考结构、关联、优先级，只需倾倒散点。AI在后台静默接收，持续扫描但不打扰。
- **Zen模式下：** 它是一个侦探的线索墙。AI已将散落的碎片初步聚拢，用户在此发现意外的关联——"Aha！"时刻在此发生。
- **贯穿始终：** 它是一棵会思考的树。用户投入种子，AI辅助生根发芽，用户在脉络图上修剪、聚焦，最终导出结构化的交付物。

### 1.4 核心价值主张

帮助创意工作者完成从"模糊念头"到"可执行方案"之间最困难的那段路：**想清楚**。

- **对个人用户：** 它是"让每一个念头，都长成它该有的样子"的创意伙伴。
- **对专业人士：** 它是"软件开发的思想原型工厂"。


## 第二部分：核心设计哲学

### 2.1 思维呼吸循环

整个产品围绕一个核心循环构建：

```
Aha模式（积累/吸气） ↔ Zen模式（涌现/呼气）
```

**Aha模式 —— 积累期**

- **场景：** 持续数天至数周的低摩擦输入期。
- **用户行为：** 随时随地，纯文本输入碎片想法。无结构、无分类、无关联要求。地铁上、会议间隙、半夜醒来。
- **AI行为：** 静默接收每一次输入，不做主动延展，不强行关联。在后台持续对新旧节点进行语义扫描，识别潜在的主题簇和弱关联，但不主动呈现。
- **用户感受：** 零负担。像把东西扔进一个永远不会满的抽屉。

**Zen模式 —— 涌现期**

- **场景：** 用户主动切换，进行整理和聚焦。
- **系统呈现：** 全屏脉络图。G6渲染引擎将AI积累的散点按潜在语义关联初步排布，用户看到一张初步编织好的网。
- **核心价值：** 用户在此刻发现AI标注的、自己从未想到的关联——"原来这两件事是相关的！"——**Aha！时刻的真正发生地。**
- **用户行为：** 手动拖拽整理，确认或断开AI建议的关联，识别主干路径，修剪废弃分支。

**即时Zen模式（v1.0 新增）**

- 即使用户只有3-5个节点，也能查看初步脉络图
- 让新用户在5分钟内体验完整循环，避免因积累期过长而流失
- 节点越少，AI关联的可信度标注越低，用户预期管理清晰

### 2.2 AI的角色与态度

AI不是被动的问答机器，而是**创意生长的引擎**。但其行为被严格约束：

| 维度 | 设计 |
|:---|:---|
| **默认角色** | 中立、务实的创意伙伴 |
| **核心行为** | 基于用户选中的节点，进行"延展"——生成多个方向的探索建议 |
| **主动与被动** | AI不主动延展，不主动批判，不主动打断。所有行为由用户触发 |
| **对抗审视** | 仅当用户点击"恶魔代言人"按钮时，AI才从批判、挑刺、寻找漏洞的角度审视 |
| **偏离检测** | 当用户输入与项目上下文偏离过大时，AI不强行关联，而是明确询问用户意图 |
| **控制权** | 完全在用户手中。用户确认才是分支，用户修剪才是废弃 |

### 2.3 根本原则

**帮助人类的创意从混沌涌现清晰，而非用AI的噪音污染人类的思考。**

### 2.4 AI 的自主权设计

AHA 对 AI 采取"插座哲学"：系统不内置任何 AI 供应商偏好，用户完全自主决定。

| 原则 | 含义 |
|:---|:---|
| **零默认供应商** | 安装后不预配置任何 AI 服务，无硬编码的 API endpoint 或模型名 |
| **协议适配而非供应商适配** | 系统对接的是通信协议（OpenAI-compatible REST API），不是具体公司 |
| **用户完全自主** | 用户自行选择供应商、配置模型、管理 API Key、控制成本 |
| **分层 AI 策略** | 用户可将不同任务分配给不同模型：延展用高质量模型，后台扫描用轻量模型，偏离检测用本地模型 |

**一句话：AHA 是插座，用户决定插什么电器。**


## 第三部分：核心工作流

### 3.1 完整工作流

**阶段一：积累期（Aha模式，持续数天至数周）**

1. 用户在不同时间、不同场景下，以纯文本输入碎片想法。
2. 每次输入被静默存储为一个节点，不做任何AI延展。
3. AI在后台持续扫描新节点与已有节点的潜在语义关联，但不主动呈现。
4. 用户完全不需要考虑结构、分类、关联。

**阶段二：涌现期（切换到Zen模式）**

1. 用户手动切换（或首次使用即时Zen模式快速预览）。
2. 系统呈现一张初步的脉络图，AI已将积累的散点按语义簇和潜在关联排布。
3. AI建议的关联以虚线显示，可信度以透明度或标签标注。
4. 用户在此刻发现意外关联，产生"Aha！"时刻。
5. 用户开始手动整理：拖动节点、确认AI建议的关联、创建新的连接。

**阶段三：生长与收敛（Aha ↔ Zen循环）**

1. 针对脉络图中的特定节点，用户可以触发AI延展（单次），选择"提取为固化分支"。
2. 用户可以随时切回Aha模式，继续倾倒新的散点。
3. 用户可以进行对抗式审查：点击"恶魔代言人"，获取批判性视角。
4. 多次循环后，用户标记主干方案，修剪废弃分支（需填写修剪原因）。

**阶段四：交付与回顾**

1. 导出实施构建包（PROJECT_CONTEXT.md + DEVELOPMENT_PACK.md）。
2. 查看Summary，看到这次跨越数天甚至数周的思考旅程的全貌。
3. Summary提供命名仪式，赋予思考以身份，闭合情感回路。

### 3.2 关键交互细节

**"确认才是分支"机制**

- 用户在节点内与AI的对话，是"草稿历史"，存储在节点内部，不向外产生新分支。
- 只有当用户对AI生成的某段延展内容长按，选择"提取为固化分支"时，才在脉络图上生成新节点和连线。
- **价值：** 确保脉络图上的每一条分支都是用户深思熟虑后的选择，而非AI的呓语。

**"修剪"的仪式**

- 用户在Zen模式下，对想放弃的分支长按，选择"修剪此分支"。
- 系统弹出确认框，**强制要求填写修剪原因**。
- 被修剪的分支被打包归档，视觉上折叠为一个点，存入L3远景记忆，可溯源但不占据认知资源。

**"恶魔代言人"按钮**

- 出现在每个节点的延展选项旁边。
- 用户点击后，AI从批判、挑刺、寻找漏洞的角度生成审视意见。
- 纯被动触发，绝不主动弹出。

**偏离检测机制**

- 当用户输入与L1核心记忆（项目宪章）偏离过大时，AI不强行延展，而是弹出询问：
  > "这看起来是一个新话题，与当前项目可能无关。你是想：A）切换项目或新建项目 B）这是个比喻，与项目有关 C）只是随口说说，不记录。"
- 用户选择C，该输入不进入项目。

| **修剪分支** | 废弃分支需填写修剪原因，归档存储 | P0 |
| **标记主干** | 将成熟脉络路径标记为"主干方案"，命名保存 | P0 |

### 4.2 后续版本规划

| 版本 | 功能 |
|:---|:---|
| v2.0 | 节点内多轮对话草稿、完整移动端体验、语音输入、可视化冲突合并工具、Anthropic/Google原生协议适配器 |
| v3.0 | L3远景记忆（LanceDB语义检索）、更丰富的AI延展类型（角色动机、情节冲突等）、Mastra记忆系统评估 |

### 4.3 已彻底废弃

| 废弃项 | 废弃原因 |
|:---|:---|
| AI主动监察模式 | 打断心流，与"用户拥有完全控制权"原则根本冲突 |
| 手写输入 | 与Git+Markdown内核不兼容，手写内容无法被Git diff和AI处理 |
| 自建P2P同步协议 | 重复造轮子，Git原生push/pull + 用户已有云盘已满足同步需求 |


## 第五部分：数据架构

### 5.1 Git原生内核

| 产品概念 | Git对应 | 说明 |
|:---|:---|:---|
| 项目 | Git仓库 | 创建项目即初始化Git仓库 |
| 创意节点 | Markdown文件 | 每个节点一个`.md`文件 |
| 脉络图结构 | `graph-index.json` | 记录节点ID、文件路径、连接关系 |
| 主干方案 | Git Tag | 用户标记主干方案即打Tag |
| 探索分支 | Git Branch | 用户创建探索方向即创建分支 |
| 修剪废弃 | `git rm` + commit | 修剪即删除文件并提交，原因写在commit message |
| 可溯源 | Git log / diff | 完整的时间线和变更对比 |
| 导出交付 | `git archive` | 打包为干净的压缩文件 |

### 5.2 分级记忆系统

| 层级 | 名称 | 存储内容 | 生命周期 | 用途 |
|:---|:---|:---|:---|:---|
| **L1** | 核心记忆 | 项目宪章、核心目标、约束条件 | 永久在线 | 每次AI交互时作为最高优先级系统指令注入，确保永不偏离项目根基 |
| **L2** | 中景记忆 | 当前选中节点及直接上下文（父节点、子节点、最近激活节点、相关主题簇） | 随用户操作动态切换 | AI延展时的思考背景 |
| **L3** | 远景记忆 | 全项目所有节点、归档分支、历史版本 | 长期持久化存储 | 通过@手动检索或AI语义检索，实现对久远想法的"回忆"（v3.0引入LanceDB） |

### 5.3 AI上下文组装机制

```
每次AI调用时，上下文组装顺序：
  1. L1核心记忆（system prompt头部，永不出户）
  2. AI人格描述（延展/恶魔代言人/关联扫描等）
  3. L2中景记忆（当前节点及直接上下文，图遍历组装）
  4. 用户当前输入
  5. (可选) L3检索结果（当用户@引用或AI自动检索到强相关时，v3.0）
```

### 5.4 数据存储结构

```
项目文件夹/
├── .git/                          # Git仓库
├── nodes/                         # 节点文件（每个创意一个.md）
│   ├── ai-huo-de-zi-you-yi-zhi.md
│   ├── jue-se-dong-ji-tuo-zhan.md
│   └── ...
├── graph-index.json               # 脉络图结构索引
├── project-constitution.md        # L1 核心记忆（项目宪章）
├── archive/                       # 归档的废弃分支
│   └── ...
└── exports/                       # 导出的实施构建包
    ├── PROJECT_CONTEXT.md
    └── DEVELOPMENT_PACK.md

全局用户配置:
  ~/.aha/
    ├── providers.json              # AI供应商列表（ProviderConfig[]）
    ├── global-routing.json         # 全局默认模型路由表
    └── usage.db                    # 用量记录（SQLite）
```

### 5.5 同步策略

- **默认方案：** 不内置同步服务器。引导用户将项目文件夹存放在云盘（iCloud、Dropbox等），或使用Git远程仓库进行手动push/pull。
- **辅助方案（v2.0+）：** 提供可视化的Markdown/JSON冲突合并工具。
- **核心原则：** 不拥有数据，只拥有体验；不管理传输，只管理冲突。


## 第六部分：用户场景

### 6.1 场景一：从零开始的创意项目（小说构思）

1. **积累期（3天）：** 用户在地铁、午餐、睡前用手机记录碎片——"AI觉醒""母子关系""股市崩盘""写诗的程序"。
2. **涌现期：** 用户打开电脑，切换到Zen模式。AI已将碎片聚类：母子关系与AI写诗被标注为强关联，股市崩盘被标记为孤立节点。
3. **生长与收敛：** 用户确认母子关系为核心主线，触发AI延展生成多个情节方向。点击"恶魔代言人"审视核心设定，发现一个逻辑漏洞。修剪掉股市分支，标记"情感主线方案"为最终主干。
4. **交付与回顾：** 导出实施构建包，内含完整的人物关系图、情节大纲和关键设定。Summary展示："你从23个碎片开始，探索了3条故事线，最终选定了以母子关系为核心的情感主线。"

### 6.2 场景二：存量项目的深度思考（代码重构）

1. **阶段零（存量理解）：** 用户导入现有项目仓库。AI对20万行代码进行静态分析，生成"代码脉络图"和架构摘要，注入L1核心记忆。
2. **发散：** 用户选中一个性能瓶颈模块，触发AI延展。AI在现有架构约束下，生成3个重构方案，每个附带了影响分析。
3. **收敛：** 用户对比方案，选择影响最小的方案B，标记为"重构主干方案"。
4. **交付：** 导出实施构建包，内含分步实施计划和风险提示。直接放入代码仓库，供AI编程助手读取。

### 6.3 场景三：长期创意管理（多个项目并行）

1. 用户同时维护3个项目：一部小说、一个产品设计、一个论文框架。
2. 每个项目拥有独立的Git仓库和分级记忆。
3. 数月后回顾任一项目，可以清晰看到每个决策的"为什么"——通过Git历史和修剪原因的记录。
4. 每次回顾时，Summary帮助用户快速重建对项目的记忆："你上次停在这里，当时你觉得B方案最有希望。"


## 第七部分：技术架构

### 7.1 技术选型

| 层 | 选型 | 说明 |
|:---|:---|:---|
| 桌面框架 | **Tauri v2** | 轻量（<10MB），跨平台，Rust后端提供原生性能 |
| 前端 | **React 18+ + TypeScript** | Web标准，一套代码支持桌面端和移动端PWA |
| 构建工具 | **Vite** | 快速HMR，与Vitest共享配置 |
| 图可视化 | **AntV G6 v5** | Canvas/WebGL多后端渲染，内置丰富的交互行为和布局算法，中文生态完善 |
| 图React封装 | **Graphin** | G6的声明式React封装 |
| Git操作 | **git2-rs（通过Tauri Command）** | Rust原生libgit2绑定，性能远优于JS实现，与Tauri架构完美契合 |
| AI SDK | **Vercel AI SDK** | 轻量，厂商无关，流式响应，`streamText` + `generateObject` 覆盖AHA全部AI场景 |
| AI供应商 | **OpenAI-compatible协议** | 事实行业标准，DeepSeek/Ollama/Groq/智谱等全部兼容，一个Adapter覆盖95%+场景 |
| AI凭据存储 | **tauri-plugin-keyring** | 跨平台系统原生密钥链（macOS Keychain / Windows Credential Manager / Linux Secret Service） |
| 向量存储 | **LanceDB**（v3.0引入） | 本地向量检索，支持L3远景记忆语义搜索 |
| 富文本编辑 | **TipTap** | 基于ProseMirror，Markdown友好，移动端兼容性最好 |
| 状态管理 | **Zustand** | ~1KB，极简API，无Provider，选择器精准订阅 |
| 撤销/重做 | **zundo** | <700B，Zustand原生中间件 |
| UI组件 | **shadcn/ui + Radix** | 可定制，无障碍，2026年React生态默认选择 |
| 动画 | **Motion (Framer Motion)** | 声明式API，layout animation，AnimatePresence处理模式切换过渡 |
| 本地缓存 | **Dexie.js (IndexedDB)** | 轻量，无需额外数据库，v1.0满足缓存需求 |
| Markdown解析 | **unified + remark + rehype** | AST管道架构，生态最全，支持格式化和安全处理 |
| 测试-单元/组件 | **Vitest + React Testing Library** | 与Vite共享配置，原生TS/ESM，~3.8x快于Jest |
| 测试-E2E | **Playwright** | 唯一同时支持WebKit+Chromium，匹配Tauri桌面端双浏览器引擎 |
| CSS | **Tailwind CSS** | 原子化CSS，与shadcn/ui配合 |
| 打包 | Tauri打包为桌面应用，PWA用于移动端 |

## 第九部分：与现有产品的核心差异

| 对比维度 | 我们的产品 | 现有接近产品 |
|:---|:---|:---|
| **数据层** | Git原生，每个节点是Markdown文件，版本管理是Git分支 | Obsidian+Git仅将Git作为备份，非核心交互 |
| **AI角色** | 创意生长引擎，受约束的延展，不主动打扰；用户完全自主配置供应商 | ChatGPT/Claude是线性对话，供应商绑定 |
| **AI供应商** | 零默认供应商，插座模式，用户自行接入任意OpenAI兼容服务 | 内置供应商偏好，用户无法自由切换 |
| **交互界面** | G6驱动的可交互生长型脉络图，AI辅助排布 | Miro/FigJam是手动白板，无AI主动延展 |
| **从散点到交付** | 积累期→涌现期→生长收敛→导出实施构建包 | 无产品提供此完整链路 |
| **版本管理** | 创意分支、标记主干方案、修剪归档，git2-rs原生性能 | Git的原生命令行门槛高，非创意用户友好 |

**我们不是在追逐一个已有的品类，而是在创造一个将"AI创意生长"、"可交互脉络图"、"Git原生版本管理"、"从散点到交付"这四个要素首次整合在一起的新交叉品类。**
